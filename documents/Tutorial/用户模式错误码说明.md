# 用户模式错误码说明

这页只解释用户模式里最常见的错误码，也就是 `@protect()` 和 `create_protector()` 失败时，`ProtectRuntimeError.code` 可能出现什么值。

如果你在代码里要显式判断错误码，推荐使用：

```python
from logicfp.user_mode import (
    ErrorCode,
    LogicExecutionError,
    NormalizationError,
    ProtectRuntimeError,
    get_error_action,
    get_error_fact,
)
```

## 1. 推荐写法

```python
from logicfp import protect
from logicfp.user_mode import ErrorCode, ProtectRuntimeError


@protect(simple=True)
def run_task(request):
    raise ValueError("manual review required")


try:
    run_task(payload={"task_id": "T-1001"})
except ProtectRuntimeError as exc:
    if exc.code == ErrorCode.ERR_NORM.value:
        print("normalization-style failure")
```

## 2. 常见错误码

### `ERR_VALIDATION`

输入数据不符合 `input_model`。

常见来源：

- `pydantic` 输入校验失败
- 必填字段缺失
- 字段类型不对

通常表现：

- `ProtectRuntimeError.details["errors"]` 里会有校验明细

### `ERR_OUTPUT_VALIDATION`

函数执行成功了，但返回结果不符合 `output_model`。

常见来源：

- 返回字段缺失
- 返回字段类型不对
- 返回结构和声明的输出模型不一致

通常表现：

- `ProtectRuntimeError.details["errors"]` 里会有校验明细

### `ERR_NULL`

函数返回了 `None`。

这通常表示：

- 你的函数没有显式 `return`
- 某个分支漏了返回值
- 逻辑上本该返回对象，但返回了空值

### `ERR_NORM`

当前版本里，下面这些 Python 异常会归到 `ERR_NORM`：

- `ValueError`
- `TypeError`
- `KeyError`

适合这些场景：

- 手动抛出“参数不合法”
- 业务前置检查失败
- 需要人工处理或输入修正的情况

如果你要稳定表达“归一化/输入修正”这类失败，建议直接抛：

```python
from logicfp.user_mode import NormalizationError
```

### `ERR_LOGIC`

当前版本里，下面这些异常会归到 `ERR_LOGIC`：

- `LogicExecutionError`
- `RuntimeError`
- `AssertionError`

更推荐的写法是显式抛：

```python
from logicfp.user_mode import LogicExecutionError
```

适合这些场景：

- 业务规则拒绝执行
- 需要人工复核
- 当前状态不允许继续处理

### `ERR_EXECUTION_BLOCKED`

请求被当前保护状态拦截，没有真正执行到业务函数。

常见场景：

- 之前发生了硬失败
- 当前状态机仍处于阻断状态
- 当前请求不是允许放行的探针请求

### `ERR_UNKNOWN`

没有被更具体规则识别的异常。

常见来源：

- 未分类的 Python 异常
- 第三方库直接抛出的异常
- 当前版本还没有归类的错误类型

如果你的代码经常出现 `ERR_UNKNOWN`，通常说明需要把这类异常做显式分类。

## 3. 其他错误码

下面这些错误码目前更多用于内部执行链：

- `ERR_TIMEOUT`
- `ERR_LOGIC`
- `ERR_HANDLER_NOT_FOUND`

用户模式里它们仍然可能出现，但普通函数保护最常见的还是前面那几类。

## 4. 错误码从哪里来

当前错误码来源主要分 4 层：

1. 输入输出校验  
   `ERR_VALIDATION` / `ERR_OUTPUT_VALIDATION`

2. 执行器兜底  
   `ERR_NULL` / `ERR_EXECUTION_BLOCKED`

3. 异常分类器  
   例如 `ValueError/TypeError/KeyError -> ERR_NORM`，`RuntimeError/AssertionError -> ERR_LOGIC`

4. 未分类异常  
   落到 `ERR_UNKNOWN`

## 5. 当前版本建议

用户模式里，最稳的判断方式是：

- 先看 `exc.code`
- 再看 `exc.details`
- 最后用 `exc.context` 关联请求上下文

如果你需要做长期稳定的错误分支处理，建议只依赖这几项：

- `ErrorCode`
- `ProtectRuntimeError.code`
- `ProtectRuntimeError.details`
- `ProtectRuntimeError.context`

## 6. 失败结构里的 4 个核心问题

如果你用的是 `simple=False`，当前版本会把失败进一步归一化成两层：

- `error.details["error_fact"]`
- `error.details["error_policy"]`

最值得依赖的是这 4 个核心字段：

- `error_fact.stage`
  - `input`
  - `execute`
  - `dependency`
  - `output`
- `error_fact.source`
  - `caller`
  - `system`
  - `dependency`
  - `environment`
  - `unknown`
- `error_fact.recoverability`
  - `retryable`
  - `degradable`
  - `non_recoverable`
  - `unknown`
- `error_policy.action`
  - `allow`
  - `warn`
  - `block`
  - `retry`
  - `fallback`
  - `trip`

也就是说：

- `error_fact` 负责回答“出了什么类型的事”
- `error_policy` 负责回答“系统默认怎么对待它”

像 `certainty`、`impact`、`user_effect`、`observability` 这类信息，当前都放在 `details` 里，更适合做诊断或内部策略，不建议当成第一层长期 contract。

如果你不想手写很多层索引，当前也可以直接用：

- `logicfp.user_mode.get_error_fact(...)`
- `logicfp.user_mode.get_error_policy(...)`
- `logicfp.user_mode.get_error_action(...)`

例如：

```python
from logicfp.user_mode import ProtectRuntimeError, get_error_action, get_error_fact

try:
    run_task(payload={"task_id": "T-1001"})
except ProtectRuntimeError as exc:
    fact = get_error_fact(exc)
    action = get_error_action(exc)
    if fact is not None:
        print(fact["stage"], fact["source"], fact["recoverability"])
    print(action)
```

如果你还想按这些事实覆写默认动作，当前高级入口是：

- `create_protector(advanced={"error_action_resolver": ...})`

它拿到的输入里会带：

- `fact`
- `default_action`

当前还会保留：

- `default_policy`

但这只是兼容字段，新代码建议优先依赖 `default_action`。

推荐返回结构是：

- `None`
  表示继续使用默认动作
- `{"action": "..."}`
  覆写默认动作

可选再补：

- `user_effect`
- `observability`
- `details`

如果你需要类型提示，当前也可以直接从 `logicfp.user_mode` 导入：

- `ErrorActionResolverPayload`
- `ErrorActionResolverResult`
- `ErrorFactData`
- `ErrorPolicyData`

最小参考在：

- `examples/user_mode/action_resolver.py`

## 7. AI 调用识别明细

如果失败来自 AI / tool / 外部依赖调用，当前版本还可能在：

- `ProtectRuntimeError.details["ai_error"]`

里带出一层更细的识别信息。

这层信息目前更偏“增强诊断”，还不是用户模式最顶层的稳定错误码，但已经可以帮助区分：

- `RATE_LIMIT_TOKEN`
- `STREAM_BROKEN`
- `AUTH_INVALID`
- `UPSTREAM_OVERLOADED`
- `OUTPUT_SCHEMA_INVALID`
- `TOOL_TIMEOUT`

典型结构会包含：

- `code`
- `category`
- `retryable`
- `severity`
- `phase`
- `provider`
- `model`
- `details`

如果你需要更强的按需识别，也可以通过：

- `create_protector(advanced={"ai_error_classifier": ...})`

传入一个自定义分类函数。它只会在规则识别没有命中时才执行。
