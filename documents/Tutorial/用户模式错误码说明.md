# 用户模式错误码说明

这页只解释用户模式里最常见的错误码，也就是 `@protect()` 和 `create_protector()` 失败时，`ProtectRuntimeError.code` 可能出现什么值。

如果你在代码里要显式判断错误码，推荐使用：

```python
from logicfp.user_mode import (
    ErrorCode,
    LogicExecutionError,
    NormalizationError,
    ProtectRuntimeError,
)
```

## 1. 推荐写法

```python
from logicfp import protect
from logicfp.user_mode import ErrorCode, ProtectRuntimeError


@protect(simple=True)
def run_task(request):
    raise ValueError("manual review required")


try:
    run_task(payload={"task_id": "T-1001"})
except ProtectRuntimeError as exc:
    if exc.code == ErrorCode.ERR_NORM.value:
        print("normalization-style failure")
```

## 2. 常见错误码

### `ERR_VALIDATION`

输入数据不符合 `input_model`。

常见来源：

- `pydantic` 输入校验失败
- 必填字段缺失
- 字段类型不对

通常表现：

- `ProtectRuntimeError.details["errors"]` 里会有校验明细

### `ERR_OUTPUT_VALIDATION`

函数执行成功了，但返回结果不符合 `output_model`。

常见来源：

- 返回字段缺失
- 返回字段类型不对
- 返回结构和声明的输出模型不一致

通常表现：

- `ProtectRuntimeError.details["errors"]` 里会有校验明细

### `ERR_NULL`

函数返回了 `None`。

这通常表示：

- 你的函数没有显式 `return`
- 某个分支漏了返回值
- 逻辑上本该返回对象，但返回了空值

### `ERR_NORM`

当前版本里，下面这些 Python 异常会归到 `ERR_NORM`：

- `ValueError`
- `TypeError`
- `KeyError`

适合这些场景：

- 手动抛出“参数不合法”
- 业务前置检查失败
- 需要人工处理或输入修正的情况

如果你要稳定表达“归一化/输入修正”这类失败，建议直接抛：

```python
from logicfp.user_mode import NormalizationError
```

### `ERR_LOGIC`

当前版本里，下面这些异常会归到 `ERR_LOGIC`：

- `LogicExecutionError`
- `RuntimeError`
- `AssertionError`

更推荐的写法是显式抛：

```python
from logicfp.user_mode import LogicExecutionError
```

适合这些场景：

- 业务规则拒绝执行
- 需要人工复核
- 当前状态不允许继续处理

### `ERR_EXECUTION_BLOCKED`

请求被当前保护状态拦截，没有真正执行到业务函数。

常见场景：

- 之前发生了硬失败
- 当前状态机仍处于阻断状态
- 当前请求不是允许放行的探针请求

### `ERR_UNKNOWN`

没有被更具体规则识别的异常。

常见来源：

- 未分类的 Python 异常
- 第三方库直接抛出的异常
- 当前版本还没有归类的错误类型

如果你的代码经常出现 `ERR_UNKNOWN`，通常说明需要把这类异常做显式分类。

## 3. 其他错误码

下面这些错误码目前更多用于工程模式、中间层或内部执行链：

- `ERR_TIMEOUT`
- `ERR_LOGIC`
- `ERR_HANDLER_NOT_FOUND`

用户模式里它们仍然可能出现，但普通函数保护最常见的还是前面那几类。

## 4. 错误码从哪里来

当前错误码来源主要分 4 层：

1. 输入输出校验  
   `ERR_VALIDATION` / `ERR_OUTPUT_VALIDATION`

2. 执行器兜底  
   `ERR_NULL` / `ERR_EXECUTION_BLOCKED`

3. 异常分类器  
   例如 `ValueError/TypeError/KeyError -> ERR_NORM`，`RuntimeError/AssertionError -> ERR_LOGIC`

4. 未分类异常  
   落到 `ERR_UNKNOWN`

## 5. 当前版本建议

用户模式里，最稳的判断方式是：

- 先看 `exc.code`
- 再看 `exc.details`
- 最后用 `exc.context` 关联请求上下文

如果你需要做长期稳定的错误分支处理，建议只依赖这几项：

- `ErrorCode`
- `ProtectRuntimeError.code`
- `ProtectRuntimeError.details`
- `ProtectRuntimeError.context`
